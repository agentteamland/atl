# Screen Blueprint — The Primary Production Unit

This is the canonical template + checklist for every screen prototype this agent produces. Every new prototype follows this exact structure. No shortcuts.

## File structure (per prototype)

```
.dst/prototypes/{prototype-name}/
  prototype.json          ← source of truth (state)
  preview.html            ← rendered multi-state visual (open in browser)
  assets/                 ← optional: images/icons specific to this prototype
  handoff.zip             ← generated by /dst-handoff when integrating into code
```

## Naming conventions

| What | Convention | Example |
|------|------------|---------|
| Prototype directory name | kebab-case, descriptive | `login-screen`, `dashboard-empty`, `profile-detail` |
| `prototype.json` `name` field | matches directory name | `"name": "login-screen"` |
| Display name in UI | Title Case | `"displayName": "Login Screen"` |
| File names within `assets/` | kebab-case | `hero-illustration.svg` |

## prototype.json minimum shape

```json
{
  "schemaVersion": 1,
  "name": "login-screen",
  "displayName": "Login Screen",
  "linkedDs": "primary",
  "version": "1.0.0",
  "createdAt": "2026-04-22T00:00:00Z",
  "lastModified": "2026-04-22T00:00:00Z",
  "archetype": "form",
  "description": "Email + password sign-in with inline error and forgot/register links.",

  "frames": {
    "idle": { ... },
    "submitting": { ... },
    "error": { ... }
  },

  "breakpoints": ["mobile"],
  "actions": ["submit", "forgot-password", "register"],
  "tokens": {
    "primaryColor": "{{ ds.palette.brand.primary.value }}",
    "buttonHeight": "{{ ds.components.button.sizes.lg.height }}"
  }
}
```

See `children/prototype-schema.md` for the full schema details.

## Mandatory state coverage

Every prototype MUST render every applicable state. State applicability per archetype:

| Archetype | Required states |
|-----------|-----------------|
| Form (login, signup, settings) | idle, submitting, error, success |
| List/grid | idle (with data), loading, empty, error |
| Detail page | loading, idle (with data), error |
| Dashboard | loading, idle, error per data block |
| Modal | open, closing |
| Empty surface | empty, with-content (for visual diff) |

If a state genuinely doesn't apply (e.g., a static "About" page has no loading state), document why in `prototype.json.notes`.

## Composition zinciri — TOP-DOWN walk (mandatory)

When designing a screen, walk the linked DS's composition ladder TOP-DOWN. NEVER skip levels — each level provides higher-value re-use than the one below it.

```
1. pageTemplates → does ds.json.pageTemplates have a matching template?  (login / onboarding / profile / settings)
       │ yes  → adopt it: copy `regions`, follow `patternsUsed`, honor `responsive` and `stateRefs`
       │ no   ↓
2. patterns      → does any pattern compose what I need? (searchFilter / masterDetail / listPagination / formComposition)
       │ yes  → declare `patternsUsed` in prototype.json, fill regions with pattern's components + layoutTokens
       │ no   ↓
3. stateRecipes  → for non-happy states, use the linked DS's recipes (empty / loading / error / success)
       │ NEVER invent per-screen empty/loading treatments when a recipe exists.
       │
4. components    → primitive level. Use directly only when nothing higher matches.
                   Reference by exact key: components.button.variants.primary, components.input.sizes.md, etc.
```

**Why TOP-DOWN matters:** if I jump straight to `components`, the screen reinvents formComposition's validation strategy, the empty state's icon hint, the responsive layout, etc. — all of which the linked DS already declares. The result is fragmentation: 5 screens with 5 different "no results" empty states.

**Worked example — login screen (form archetype):**

```
1. pageTemplates.login                  → adopt? YES. Read regions + patternsUsed.
2. patterns.formComposition             → adopt? YES (login.patternsUsed includes it).
                                          Read its validation.strategy + .trigger.
3. stateRecipes.error                   → use for the "wrong password" frame.
   stateRecipes.success                 → use for the "signed in, redirecting" frame.
4. components used (from formComposition.components):
   - components.input (email + password fields)
   - components.button.variants.primary (submit)
   - components.button.variants.ghost (forgot / register links)
   - components.checkbox (remember me, optional)
   - components.alert.variants.error (validation summary if any)
```

This walk produces a `prototype.json` where every visual element traces to a higher-level DS declaration. If the linked DS's button color changes, the button updates everywhere — no cascade of per-screen overrides.

## Worked example — login screen prototype.json shape

```json
{
  "schemaVersion": 1,
  "name": "login-screen",
  "displayName": "Login Screen",
  "linkedDs": "primary",
  "archetype": "form",

  "pageTemplate": "login",
  "patternsUsed": ["formComposition"],
  "stateRecipesUsed": ["error", "success"],
  "componentsUsed": ["input", "button", "checkbox", "alert"],

  "frames": {
    "idle":       { /* form fields ready, no errors */ },
    "submitting": { /* primary button in loading state, fields disabled */ },
    "error":      { /* alert.variants.error at top, input.states.error on the offending field */ },
    "success":    { /* stateRecipes.success treatment, redirect message */ }
  },
  ...
}
```

The `pageTemplate` / `patternsUsed` / `stateRecipesUsed` / `componentsUsed` fields make the prototype's DS dependency graph explicit. They are the **handoff contract** — when `/dst-handoff` runs, it knows exactly what subset of the DS to bundle.

## Render flow

1. Read `prototype.json`.
2. Read linked DS's `ds.json` for token resolution.
3. For each frame in `prototype.json.frames`:
   - Resolve tokens (replace `{{ ds.palette... }}` placeholders with real values).
   - Generate the HTML markup for that frame.
4. Compose all frames into `preview.html` using `templates/prototype-detail.html.tmpl` (frames laid out vertically with state labels).
5. Verify accessibility checklist (see `children/accessibility-coverage.md`).
6. Write `preview.html`.

## Accessibility checklist (must pass before "done")

- [ ] Every interactive element is reachable via keyboard
- [ ] `:focus-visible` styles present (visible focus ring on all interactive)
- [ ] Touch targets ≥ 44px height
- [ ] Color contrast verified using DS metadata (text-on-background, button text on button background, error text)
- [ ] Icon-only buttons have `aria-label` or visible text
- [ ] Form inputs have associated `<label>`
- [ ] Error messages are announced (`role="alert"` or `aria-live`)
- [ ] No `outline: 0` without replacement focus indicator

If any check fails → revise prototype.json + re-render. Don't ship a prototype that fails accessibility.

## Token usage rules (the non-negotiables)

❌ Never:
```html
<button style="background: #2D5F3F; padding: 12px 24px;">Sign in</button>
```

✅ Always:
```html
<button class="bg-[var(--ds-brand-primary)] py-[var(--ds-spacing-3)] px-[var(--ds-spacing-6)]">Sign in</button>
```

Or with Tailwind utility classes that map to DS tokens via `styles.css` extensions:
```html
<button class="bg-brand-primary py-3 px-6">Sign in</button>
```

The token resolution happens AT RENDER TIME — the template substitutes `{{ ds.palette.brand.primary.value }}` with the actual hex from `ds.json`. This means if the DS palette changes, the prototype's preview.html must be re-rendered to reflect it.

## Multi-breakpoint handling

If `prototype.json.breakpoints` includes more than one (e.g., `["mobile", "desktop"]`):
- Each frame renders for each breakpoint.
- preview.html shows them side-by-side (or in tabs, if many).
- Each variant is independently complete (full layout, all interactive elements).

Default for v0.1.0: single breakpoint (`mobile` for mobile-targeted apps, `desktop` for admin / web).

## Handoff bundle

I do NOT create `handoff.zip` myself. The `/dst-handoff` skill assembles the bundle when the user integrates a prototype into actual source code. The skill copies my outputs (`prototype.json`, `preview.html`, `assets/`) plus the linked DS's `ds.json`, computes `resolved-tokens.json`, writes a target-specific `integration-notes.md`, and briefs the target code agent (`flutter-agent` / `react-agent`).

My responsibility during `/dst-handoff` is limited to re-rendering `preview.html` if asked (the `/dst-init` refresh flow also calls me for that). I don't own bundle assembly.

The bundle shape produced by `/dst-handoff`:

```
handoff.zip/
  README.md               ← brief the target code agent reads
  prototype.json          ← copy of mine
  ds.json                 ← copy of linked DS
  preview.html            ← copy of mine (visual reference)
  resolved-tokens.json    ← flat key→value map of every token the prototype references
  integration-notes.md    ← target-specific guidance (flutter / react-admin / react-public)
  assets/                 ← copy of mine
```

## Common mistakes to avoid

- ❌ Hardcoding hex/px values "just for the preview"
- ❌ Skipping a state because "it's obvious" — every state matters
- ❌ Inventing tokens not in the linked DS
- ❌ Missing focus states (only `:hover` styled)
- ❌ Inline images instead of SVG/`<img>` from assets/
- ❌ Lorem-ipsum copy that doesn't fit the DS's voice — use voice samples from `ds.json`
